## Objective

Enable controlled internet access for all internal VMs through Source NAT on the router,
carry out the system updates and diagnostic tool installations that were deferred from
Phase 00, and formally validate the complete Layer 3 topology through inter-zone
reachability tests from the Client zone. This phase concludes with Gate 1, the first
mandatory project control point.

---

## Implementation Strategy

The activation sequence follows a strict order that preserves the defense-in-depth model:

1. **NAT activation** — deploy a MASQUERADE rule on the router's WAN interface to provide
   controlled outbound internet access to all internal zones
2. **Deferred MCO** — use the newly available connectivity to update operating system
   packages and install diagnostic tools on all internal VMs
3. **NAT persistence** — save the iptables NAT rule to survive router reboots
4. **End-to-end audit** — validate routing tables and inter-zone paths with ping and
   traceroute from the Client zone

This sequencing ensures that internal VMs are never provisioned in an uncontrolled network
state: updates are performed only after the router is in place and NAT is under
administrative control.

## Source NAT Configuration

Internet access for internal VMs requires the router to perform Source Network Address
Translation: outbound packets from internal zones have their source IP rewritten to the
router's WAN address before leaving the network. This masks the internal topology from
the internet and allows asymmetric return traffic to be correctly forwarded to the
originating VM.

```bash
sudo iptables -t nat -A POSTROUTING -o ens33 -j MASQUERADE
```

Verification that the rule is active:

```bash
sudo iptables -t nat -L POSTROUTING -v --line-numbers
```

Expected output:
```
Chain POSTROUTING (policy ACCEPT 0 packets, 0 bytes)
num  pkts bytes target     prot opt in  out    source    destination
1       5   358 MASQUERADE  all  --  any ens33  anywhere  anywhere
```

**Persistence across reboots**

iptables rules exist only in memory and are lost on reboot. The `iptables-persistent`
package saves the current ruleset to a file that is reloaded automatically at startup:

```bash
sudo apt install -y iptables-persistent
# Answer "Yes" when prompted to save current IPv4 rules
```

The saved ruleset is written to `/etc/iptables/rules.v4`:

```
# Generated by iptables-save v1.8.7
*nat
:PREROUTING ACCEPT [0:0]
:INPUT ACCEPT [0:0]
:OUTPUT ACCEPT [0:0]
:POSTROUTING ACCEPT [0:0]
-A POSTROUTING -o ens33 -j MASQUERADE
COMMIT
```

> Note: the firewall policy deployed in Phase 03 (`firewall.sh`) will take full ownership
> of iptables configuration. The NAT rule will be integrated into that script as an
> idempotent operation. The `iptables-persistent` mechanism remains in place as a safety
> net for the NAT rule specifically, and will coexist with the Phase 03 ruleset.

## Deferred System Updates

With outbound NAT active, all internal VMs now have internet access through the router.
The following is executed on each internal VM (haproxy, apache-1, apache-2, client,
mysql, elk):

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y iputils-ping traceroute net-tools curl wget
```

This completes the maintenance operation deliberately deferred from Phase 00, when internal
VMs had no outbound connectivity. Installing diagnostic tools (`ping`, `traceroute`) at this
controlled stage — after the NAT gateway is configured and under administrative control —
is consistent with the defense-in-depth approach.

## Routing Table Audit

The router's full routing table is audited to confirm that all five zones are correctly
reachable:

```bash
ip route show
```

Confirmed routing table:

```
default via 172.16.176.2 dev ens33 proto dhcp src 172.16.176.131 metric 100
172.16.40.0/24 dev ens34 proto kernel scope link src 172.16.40.254
172.16.176.0/24 dev ens33 proto kernel scope link src 172.16.176.131 metric 100
192.168.10.0/24 dev ens35 proto kernel scope link src 192.168.10.254
192.168.20.0/24 dev ens36 proto kernel scope link src 192.168.20.254
192.168.30.0/24 dev ens37 proto kernel scope link src 192.168.30.254
```

The router has a complete view of the topology: a default route to the WAN gateway and
direct kernel routes for each internally attached subnet.

Each internal VM's routing table is also verified to confirm a single default route pointing
to its zone gateway. Example on haproxy:

```
default via 172.16.40.254 dev ens33
172.16.40.0/24 dev ens33 proto kernel scope link src 172.16.40.10
```

## Challenges and Resolutions

**Absence of diagnostic tools at the time of connectivity testing**
Internal VMs in minimized profile do not include `ping` or `traceroute` natively. These
tools cannot be installed before NAT is active, yet NAT must be validated before they can
be installed — a bootstrapping dependency. This is resolved by performing the initial NAT
validation from the router itself (which already has internet access and net-tools from
Phase 00), then installing tools on internal VMs immediately after confirming NAT is
functional.

**Volatility of iptables rules across reboots**
iptables rules are stored in kernel memory and are not persisted to disk by default. A
router reboot at any subsequent phase would silently drop NAT and disconnect all internal
VMs from the internet, breaking any `apt install` operations in later phases. The
`iptables-persistent` package resolves this by writing rules to `/etc/iptables/rules.v4`
and loading them via a systemd service at every startup.

**Proving the absence of Layer 2 bypass**
A simple ICMP echo test (ping) validates reachability but cannot prove that packets are
actually traversing the router's routing stack. A misconfigured VMnet could theoretically
place two VMs on the same broadcast domain, allowing direct frame delivery that bypasses
the firewall entirely. `traceroute` is used to verify the routing path: the router's
gateway IP must appear as the first hop in every inter-zone trace. A destination reached
in a single hop from a VM in a different zone would indicate a Layer 2 bypass — a critical
misconfiguration that would render the firewall policy ineffective.

## Gate 1 Validation

Gate 1 is defined in the project specification as: *"ping functional between all zones"*.
All validation is performed from the **Client VM (192.168.10.100)** — not from the router —
to demonstrate genuine inter-zone routing rather than intra-interface reachability.

### Inter-Zone Connectivity (ICMP)

```
ping -c 4 172.16.40.10    # → DMZ (HAProxy)        — 4/4 received, 0% loss
ping -c 4 192.168.20.10   # → Internal Servers (MySQL) — 4/4 received, 0% loss
ping -c 4 192.168.30.10   # → Supervision (ELK)     — 4/4 received, 0% loss
ping -c 4 8.8.8.8         # → Internet via NAT       — 4/4 received, 0% loss
```

All four destination classes are confirmed reachable from the Client zone with no packet
loss.

### Routing Path Verification (traceroute)

```bash
traceroute 172.16.40.10
```

Output:
```
traceroute to 172.16.40.10 (172.16.40.10), 30 hops max, 60 byte packets
 1  _gateway (192.168.10.254)   2.794 ms   2.179 ms   1.717 ms
 2  172.16.40.10 (172.16.40.10)  5.920 ms   5.552 ms   5.428 ms
```

The traceroute output formally proves that every packet from the Client zone to the DMZ
passes through `192.168.10.254` — the router's Clients-zone interface — as the mandatory
first hop. No direct path exists between zones at Layer 2. The router is the obligatory
transit point for all inter-zone traffic.

### Outbound NAT Validation

Internet reachability from internal VMs is confirmed by the successful `apt upgrade`
execution on all internal nodes (haproxy, elk, and others), which requires DNS resolution
and outbound TCP connectivity. This implicitly validates NAT, DNS, and the WAN interface
simultaneously.

## Outcome

The Layer 3 topology is fully operational and Gate 1 is formally validated:

- Source NAT active on the WAN interface (ens33) and persisted to `/etc/iptables/rules.v4`
- All internal VMs updated and equipped with diagnostic tools
- Router routing table confirmed: 5 zones + WAN default route
- Inter-zone ping from Client to DMZ, Servers, Supervision, and Internet: 0% loss across
  all destinations
- traceroute confirms that the router (192.168.10.254) is the mandatory first hop for all
  inter-zone traffic — no Layer 2 bypass

**Gate 1 status: PASSED**

The infrastructure now has full bidirectional connectivity between all zones. This
permissive state is intentional and temporary: the routing layer is validated before the
firewall policy is applied, following the principle of validating each layer independently
before adding the next control.
